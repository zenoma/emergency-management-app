\chapter{Análisis de viabilidad}
\label{chap:viabilidade}

\lettrine{E}{n este capítulo} se analiza la viabilidad del proyecto desde tres perspectivas complementarias: dimensión temporal, dimensión tecnológica y dimensión económica. 

\section{Viabilidad temporal}
Para la realización de este proyecto se ha estimado una duración aproximada de cuatro meses. Dicha planificación incluye las fases de análisis, diseño, implementación, pruebas y redacción de la memoria. Además, se contempla la familiarización de las nuevas tecnologías. Dado el alcance definido, la viabilidad temporal es aceptable y no representa un riesgo crítico para la finalización.

La metodología usada utiliza desarrollos cortos para ir validando el progreso y ajustando el alcance si fuera necesario.

\section{Viabilidad tecnológica}
Para el estudio de viablidad tecnológica se han considerado las tecnologías heredadas del trabajo previo \cite{TFGAdrianGarcia2025}, así como las incorporaciones necesarias para completar la parte móvil y el sistema de avisos.

En resumen, la solución se apoya en un backend para servicios REST y lógica de negocio, integrado con una base de datos espacial que permita gestionar cuadrantes, ubicaciones y recursos georreferenciados.

La interfaz web mantiene una arquitectura orientada a la gestión operativa, mientras que la parte móvil se resuelve con una aplicación nativa pensada para acceder a sensores del dispositivo y mostrar la información de forma cómoda en pantalla.

Por último, el sistema de notificaciones push permite entregar avisos fiables a los dispositivos Android en tiempo real.

En lo relativo de Kotlin y \acrshort{fcm}, ambas tecnologías están ampliamente adoptadas, cuentan con extensa documentación, y existen guías oficiales para su integración con backends Java/Spring. Por tanto, su adopción no supone un riesgo tecnológico significativo, aportan ventajas en productividad, interoperabilidad móvil y soporte nativo para notificaciones.

Las demás tecnologías son conocidas por el autor gracias a su dedicación profesional y al proyecto previo, lo que reduce el riesgo de adopción y acelera el desarrollo.


\section{Viabilidad económica}
El proyecto no requiere una inversión económica significativa en hardware o licencias de software. El software utilizado es mayoritariamente Open Source (Spring Boot, PostgreSQL, React, Kotlin/Jetpack Compose), por lo que no hay costes de licencia en el entorno de desarrollo académico.

En el ámbito del desarrollo y las pruebas estos pueden ejecutarse con recursos disponibles por lo que no es necesario adquirir nuevo hardware. Sobre el uso de servicios en la nube o APIs de terceros (\acrshort{fcm}, MapLibre, OpenWeatherMap) pueden surgir costes, pero se pueden controlar gracias a los planes gratuitos o a la monitorización de uso.

La dimensión económica no supone un riesgo significativo para la ejecución del proyecto, pues los costes directos son bajos.

\section{Conclusión del análisis de viabilidad}
En resumen, el proyecto es totalmente viable porque combina un plan de trabajo realista de cuatro meses con un uso de la tecnología. Al reutilizar la base del trabajo previo \cite{TFGAdrianGarcia2025} y apoyarse en herramientas gratuitas y potentes como Kotlin y \acrshort{fcm}, el riesgo técnico es mínimo y los costes se mantienen relativamente bajos. Esta experiencia previa, sumada a una planificación clara, garantiza que el proyecto se pueda terminar a tiempo y con la calidad necesaria.
