\chapter{Diseño}
\label{chap:deseño}

\lettrine{E}{n} este capítulo se describe la organización interna de la aplicación y las principales decisiones de diseño adoptadas en sus tres subsistemas: backend, frontend y Android. El objetivo ha sido mantener una estructura coherente con el dominio funcional del proyecto, facilitar el mantenimiento del código y conservar una separación clara entre persistencia, lógica de negocio, exposición de servicios e interfaz de usuario.

\section{Introducción y objetivos}

El diseño parte de una idea principal: el sistema debe permitir coordinar emergencias, recursos y avisos de forma consistente, sin duplicar responsabilidades entre capas. Para ello, el backend concentra el modelo de dominio y las reglas de negocio; el frontend web actúa como interfaz de gestión principal; y la aplicación Android adapta parte de esas funcionalidades a un contexto móvil, manteniendo el acceso a la información operativa y a las acciones más frecuentes.

En la evolución respecto al \acrshort{tfg} se han incorporado nuevas funcionalidades, especialmente en torno a las asignaciones, envío de notificaciones y una generación de recomendaciones basadas en reglas de recomendación. Sin embargo, la estructura general de la interfaz y del dominio se mantiene casi intacta, porque el objetivo no era redefinir la estructura de la aplicación completa sino ampliar su alcance funcional sobre una base ya establecida. Gracias a que existía una base escalable y bien organizada, se han podido añadir nuevas funcionalidades sin necesidad de reestructurar el código existente, lo que ha facilitado la evolución del proyecto.

\section{Resumen de patrones usados}
Parte de la arquitectura y de las decisiones de diseño utilizadas en este trabajo han sido heredadas y adaptadas a partir del \acrshort{tfg} desarrollado previamente. Esta reutilización no solo permite mantener una continuidad tecnológica entre ambos proyectos, sino también reducir la complejidad de desarrollo al apoyarse en una base arquitectónica ya consolidada.

Entre los patrones heredados destaca especialmente la organización en capas, utilizada tanto en el backend como en el frontend, permitiendo separar claramente las responsabilidades de presentación, lógica de negocio y persistencia de datos. Esta división facilita el mantenimiento del código, mejora la modularidad y simplifica la evolución futura de la aplicación. Del mismo modo, se mantiene una estructura basada en el patrón Modelo-Vista-Controlador (MVC), especialmente en la parte de interfaz, favoreciendo una separación clara entre la representación visual, la gestión del estado y la interacción con el usuario.

También se utilizan mecanismos de abstracción y desacoplamiento como la inyección de dependencias, que facilita la reutilización de componentes y reduce el acoplamiento entre módulos, o el uso de objetos específicos para la transferencia de información entre capas, permitiendo encapsular y estructurar adecuadamente los datos intercambiados dentro del sistema. A mayores, se conservan patrones orientados al acceso y manipulación de datos, contribuyendo a aislar la lógica de persistencia del resto de la aplicación y mejorando la mantenibilidad general del proyecto.

Junto a estos patrones heredados, se han incorporado otros vinculados a la lógica de recomendación del backend y a la comunicación asíncrona y navegación de la aplicación Android. A continuación se resumen los patrones añadidos.

\begin{itemize}

    \item \textbf{API-first con OpenAPI:} En este proyecto se ha adoptado un diseño \textit{API-first}, definiendo primero el contrato de la API en un fichero YAML con OpenAPI y generando a partir de él parte de los controladores y los DTOs del backend mediante el plugin de Maven de OpenAPI. Esta forma de trabajo permite que el frontend web y la aplicación Android consuman una interfaz ya pactada, reduciendo discrepancias entre la implementación y la documentación.

    \item \textbf{Chain of Responsibility:} En el backend, la lógica de recomendaciones se apoya en este patrón, ya que las reglas se evalúan de forma secuencial hasta encontrar la primera que cumple la condición buscada. Cada regla actúa como un eslabón de la cadena y, cuando satisface el criterio, devuelve la propuesta asociada sin continuar con las siguientes.

    \item \textbf{Observer:} En Android, la integración con \textit{Firebase Messaging} se ajusta a este patrón, porque el dispositivo se suscribe mediante su token FCM y recibe notificaciones cuando el backend emite eventos relevantes. Esta comunicación por eventos permite desacoplar el emisor de la notificación del receptor.

    \item \textbf{Jetpack Compose:} En la parte Android se ha adoptado este enfoque declarativo para la interfaz, junto con una navegación por destinos mediante \textit{NavHost}. La aplicación sigue además un diseño de cliente ligero, apoyándose en el backend para la lógica de negocio y reservando para el móvil la composición de pantallas, la navegación y la interacción con el usuario.

\end{itemize}

\section{Arquitectura general}

La aplicación se organiza como un sistema cliente-servidor con tres subsistemas diferenciados. El backend implementa la lógica central del dominio y expone una \acrshort{api} \acrshort{rest}; el frontend web consume esa \acrshort{api} como una \acrshort{spa}; y Android actúa como cliente móvil independiente, reutilizando el mismo backend.

Esta división permite que cada subsistema evolucione con una responsabilidad clara. El backend no depende de la tecnología de interfaz, el frontend web no contiene reglas de negocio relevantes y la aplicación Android adapta el consumo de datos a su navegación y a sus pantallas móviles.

El modelado utilizado en esta memoria sigue una perspectiva C4 \cite{C4Model}, comenzando por una vista de contexto que sitúa el sistema en su entorno y continuando con vistas de contenedor para cada uno de los subsistemas principales.

\section{Subsistema Backend}

\subsection{Objetivos}

El backend tiene como objetivo coordinar el estado de las emergencias, recursos, asignaciones y avisos. Debe exponer una \acrshort{api} \acrshort{rest} que permita al frontend web y a la aplicación Android consultar información operativa, gestionar emergencias, crear asignaciones, enviar avisos y configurar reglas de recomendación. Además, el backend debe garantizar la seguridad mediante autenticación y autorización, así como registrar históricos de todas las operaciones que se realicen para trazabilidad y auditoría.

\subsection{Arquitectura}

La estructura del backend sigue un esquema clásico de Spring Boot. En la capa de entrada se encuentran los controladores \textit{REST}, que reciben las peticiones y devuelven \textit{DTOs}. En la capa intermedia se sitúan las fachadas y servicios, donde se implementa la lógica de negocio. En la capa inferior están las entidades JPA y los repositorios, responsables de la persistencia.

\subsection{Modelo del dominio}

\subsubsection{Diagrama de entidades}

El modelo de dominio se ha ampliado de forma notable respecto al \acrshort{tfg}. La figura \ref{fig:modelo-dominio} resume la estructura persistente actual y permite observar cómo el dominio ha pasado de estar centrado en incendios, recursos y avisos a incorporar de forma explícita emergencias, asignaciones, reglas de recomendación y gestión de históricos más detallada.

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{imaxes/modelo_dominio.png}
  \caption{Diagrama de entidades del modelo de dominio}
  \label{fig:modelo-dominio}
\end{figure}

En la figura \ref{fig:modelo-dominio} se observan las entidades persistentes principales y sus relaciones. Frente al \acrshort{tfg}, la entidad \textit{Fire} ha sido sustituida por \textit{Emergency}, que amplía el concepto de incidencia y mantiene atributos como descripción, tipo, índice, fechas de creación y resolución, además de admitir tanto localización por punto como vinculación a cuadrantes. Esta ampliación refleja mejor el funcionamiento real de la aplicación, ya que ahora una emergencia puede gestionarse por cuadrantes o por coordenadas.

Otra diferencia relevante es la aparición de \textit{EmergencyQuadrant}, que conserva la relación histórica entre emergencias y cuadrantes con su fecha de vinculación. En el \acrshort{tfg} esta información ya existía vinculada al incendio, pero en el \acrshort{tfm} se ha decidido darle una entidad propia para reflejar mejor la evolución de las emergencias a lo largo del tiempo y su relación con los cuadrantes. 

Las asignaciones pasan a tener una entidad propia, \textit{Assignment}, con mayor alcance que en el proyecto anterior. Ahora no solo enlazan recursos con una emergencia, sino que también pueden referirse a un cuadrante concreto o a una emergencia basada en punto, almacenando además su estado, observaciones y marcas temporales del ciclo de vida. Esta ampliación es clave para reflejar los recursos utilizados en las resoluciones de las emergencias.

Los recursos siguen compartiendo la clase base \textit{Resource}, pero su alcance funcional se ha ampliado. \textit{Team} y \textit{Vehicle} heredan de ella y, además de sus datos específicos, ahora participan de forma explícita en el ciclo de despliegue y retirada a través de estados, fechas de despliegue y desmantelamiento y asociaciones con asignaciones. Esto refuerza el papel operativo de ambos tipos de recurso dentro de la aplicación.

En la parte de personal, \textit{User}, \textit{Organization} y \textit{OrganizationType} mantienen la estructura general del \acrshort{tfg}, pero su uso se ha ampliado con nuevas relaciones operativas, como la pertenencia a equipos y el registro de dispositivos móviles mediante \textit{MobileDevice}. También se refuerza el papel de \textit{Organization} como contenedor de recursos y unidad de gestión. El bloque de avisos conserva la idea del proyecto anterior con \textit{Notice} e \textit{Image}.

Finalmente, la parte de históricos ya no se apoya únicamente en registros específicos por tipo de recurso, sino en \textit{AssignmentLog}, que unifica la trazabilidad de eventos relacionados con emergencias, cuadrantes y recursos. Esta evolución simplifica la consulta del pasado operativo del sistema y aporta una visión más homogénea de los cambios producidos en la gestión diaria.

\subsubsection{Modelo de datos}

El modelo de datos también se ha ampliado de forma notable respecto al \acrshort{tfg}. La figura \ref{fig:diagram-bd} resume el esquema relacional actual y permite ver cómo la base de datos ha pasado de una estructura centrada en incendios, recursos y registros históricos separados por tipo a un modelo más homogéneo, orientado a emergencias, asignaciones y trazabilidad operativa.

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{imaxes/diagram-bd.png}
  \caption{Diagrama relacional de la base de datos}
  \label{fig:diagram-bd}
\end{figure}

En la figura \ref{fig:diagram-bd} se observa que el modelo relacional sigue reproduciendo de forma directa el modelo de dominio, pero con relaciones y tablas adicionales que amplían el alcance funcional del sistema. Frente al \acrshort{tfg}, la aparición de la tabla de \textit{assignment} introduce una entidad central para vincular recursos con emergencias y cuadrantes, mientras que \textit{emergency\_quadrant} consolida la relación entre emergencias y cuadrantes como una relación con histórico propio. Además, se incorpora \textit{emergency\_type\_rule} para almacenar las reglas que alimentan el motor de recomendación, y \textit{mobile\_device} para asociar usuarios con el token FCM utilizado en notificaciones móviles.

También se mantienen las tablas de organizaciones, usuarios, recursos, avisos e imágenes, pero su uso queda ahora integrado en un dominio más amplio. En particular, el antiguo bloque de históricos por recurso se reorganiza alrededor de \textit{assignment\_log}, lo que unifica el registro de eventos y facilita la consulta del pasado operativo. Por último, el modelo conserva campos geoespaciales para almacenar puntos, polígonos y multipolígonos, ya que el sistema sigue trabajando con cuadrantes, localizaciones de emergencias y localizaciones de organizaciones.


\subsubsection{Consideraciones adicionales de modelado de datos}

La mayoría de entidades siguen utilizando claves primarias autogeneradas, lo que simplifica la persistencia y evita lógica manual de asignación de identificadores. Las relaciones entre entidades se definen con anotaciones JPA, principalmente \textit{@ManyToOne}, \textit{@OneToMany} y \textit{@OneToOne}, según la cardinalidad real del dominio.

Respecto al \acrshort{tfg}, se mantiene la política de carga perezosa en las relaciones que pueden crecer en tamaño, como usuarios de un equipo o cuadrantes de una emergencia, pero ahora esa decisión cobra más importancia porque el número de asociaciones ha aumentado. En particular, las relaciones de asignaciones, logs y reglas de recomendación añaden nuevas rutas de navegación entre entidades que conviene cargar solo cuando resultan necesarias.

También se han reforzado las validaciones de Bean Validation y las restricciones funcionales derivadas del propio modelo. Esto afecta tanto a entidades ya existentes, como \textit{Resource} o \textit{User}, como a las nuevas entidades relacionadas con asignaciones y recomendaciones. En \textit{Resource} se centralizan pequeñas reglas de coherencia, por ejemplo el borrado de \textit{deployAt} cuando el recurso vuelve al estado disponible, lo que reduce la dispersión de lógica repetida.

Otro cambio relevante respecto al \acrshort{tfg} es que las relaciones históricas se han vuelto más precisas y homogéneas. El proyecto anterior separaba parte de la trazabilidad por tipos de recurso; en el \acrshort{tfm}, en cambio, el modelo de asignaciones y logs unifica esa información y permite consultar de forma más clara qué recurso estuvo vinculado a qué emergencia, en qué cuadrante y en qué momento.

\subsection{Capa de acceso a datos}

La capa de acceso a datos se implementa mediante repositorios que extienden \textit{JpaRepository}. Esto proporciona operaciones CRUD sin necesidad de escribir consultas para los casos más comunes. Cuando hace falta un acceso más específico, se añaden métodos derivados por nombre o consultas personalizadas.

Un ejemplo claro es \textit{QuadrantRepository}, que además de búsquedas por escala o por emergencia incluye una consulta geoespacial para calcular superficies en hectáreas. También hay repositorios específicos para entidades como asignaciones, históricos, usuarios, recursos y avisos. Este enfoque permite mantener la persistencia muy cerca del modelo relacional, pero sin perder flexibilidad cuando el dominio necesita consultas particulares.


\subsection{Capa de lógica de negocio}

La lógica de negocio se organiza en \textit{facades} e interfaces de servicio. \textit{PersonalManagementFacade} agrupa operaciones de organizaciones y usuarios; \textit{ResourceManagementFacade} agrupa equipos y vehículos; \textit{EmergencyManagementService} centraliza las operaciones sobre emergencias y cuadrantes; \textit{AssignmentService} gestiona el ciclo de vida de las asignaciones; y \textit{LogManagementService} se encarga de registrar eventos históricos.

En la fachada de personal se gestionan las operaciones de registro, inicio de sesión, edición de perfil, cambio de contraseña, asociación de dispositivos móviles y modificación de roles. En la fachada de recursos, se permiten altas, modificaciones, bajas lógicas y gestión de miembros en equipos. Las organizaciones también se crean y se actualizan desde esta capa.

La parte de emergencias concentra las operaciones de creación, actualización, resolución y vinculación a cuadrantes o puntos. La lógica verifica restricciones de negocio como no modificar una emergencia resuelta o no vincular una emergencia a un punto si ya está asociada a cuadrantes.

Las asignaciones son el núcleo más dinámico del sistema. \textit{AssignmentServiceImpl} comprueba si el recurso está ocupado, si la emergencia existe, si la emergencia ya fue resuelta y si el cuadrante pertenece realmente a la emergencia. Al crear una asignación, registra eventos en históricos y envía notificaciones al \acrshort{fcm}. Cuando una asignación cambia de estado, el servicio actualiza el recurso asociado, marca la fecha de despliegue o retirada y vuelve a registrar el evento correspondiente. En caso de resolución de la emergencia, las asignaciones pendientes se eliminan lógicamente y las aceptadas se completan automáticamente.

La recomendación semiautomática de recursos se apoya en \textit{EmergencyTypeRule} y en un motor de reglas que interpreta un JSON con una parte \textit{when} y una parte \textit{then}. El servicio de edición valida que el JSON sea correcto antes de persistirlo. El motor de evaluación compara el tipo de emergencia con las condiciones de cada regla y devuelve una propuesta con número de equipos, vehículos y parámetros adicionales como distancia máxima u organización preferente.

\subsection{Capa servicios}

\subsubsection{Diseño de controladores}

Una de las diferencias más importantes respecto al \acrshort{tfg} es que los controladores se generan a partir de la especificación openapi. En el proyecto anterior la definición de endpoints dependía de clases controladoras creadas manualmente, mientras que en el \acrshort{tfm} se parte de la descripción de la API y el código generado proporciona la base de los controladores y de los contratos de entrada y salida.

Este enfoque reduce errores de sincronización entre documentación e implementación, y facilita que los cambios en la API se propaguen de forma consistente. Los controladores siguen siendo responsables de exponer la operación HTTP, pero la estructura general, los tipos de petición y respuesta y parte de la firma de los endpoints quedan alineados con el contrato definido en OpenAPI.

  \includegraphics[width=0.95\textwidth]{imaxes/swagger.png}
  \caption{Colección de endpoints documentados en Swagger}

En la figura \ref{fig:swagger-collection} se muestra la colección de endpoints expuestos por la API y documentados mediante Swagger. Esta vista permite comprobar de forma centralizada los recursos disponibles, los métodos HTTP asociados y la estructura general de la interfaz pública del backend.

\subsubsection{Gestión de errores e internalización}

La gestión de errores se centraliza en \textit{GlobalControllerExceptionHandler}. Allí se traducen excepciones de instancia, de dominio y de validación a respuestas HTTP adecuadas y mensajes localizados. Este mecanismo es heredado del \acrshort{tfg} y apenas ha necesitado modificaciones estructurales, ya que sigue siendo útil para homogeneizar la respuesta ante errores y para mantener la internacionalización de mensajes.

\subsubsection{Control de acceso}

La configuración de seguridad se define en \textit{SecurityConfig}. El backend funciona con sesión sin estado y filtro JWT. Se autorizan explícitamente los endpoints según rol: usuarios anónimos pueden iniciar sesión, registrarse y consultar parte de la información pública; los roles de miembro, gestor y coordinador acceden a recursos y emergencias según permisos; y las operaciones de coordinación quedan restringidas a coordinadores.

Este bloque también es heredado del \acrshort{tfg} y se ha mantenido prácticamente sin cambios estructurales, porque la política de permisos seguía siendo válida para la evolución de la aplicación. La aplicación también registra dispositivos móviles mediante \textit{MobileDevice}, lo que permite enlazar el usuario con su token FCM y habilitar notificaciones push.

\subsection{Librerías y servicios externos}

Además de la persistencia y la seguridad, el backend integra varias librerías y servicios externos. Se usa PostGIS para operaciones geoespaciales, Jackson para el manejo del JSON de reglas de recomendación, y \acrshort{fcm} para notificaciones móviles. También se apoya en una capa de mapeo entre entidades y \textit{DTOs} para desacoplar la API pública del modelo interno.

\section{Subsistema Frontend}

\subsection{Objetivos}

El frontend web ofrece la interfaz principal de gestión y consulta. Respecto al \acrshort{tfg}, no se han producido cambios sustanciales en la arquitectura general, sino una ampliación de vistas y funcionalidades que se ha apoyado en la base ya existente. La aplicación debe seguir permitiendo autenticación, navegación por módulos, visualización de emergencias sobre mapa, gestión de recursos, asignaciones, avisos y reglas de recomendación, además de adaptarse a distintos roles sin romper la experiencia de uso.

La aplicación se ha mantenido como una \acrshort{spa} para evitar recargas completas de página y facilitar una interacción continua con mapas, tablas y formularios. Esta decisión sigue siendo adecuada porque permite ampliar pantallas y flujos sin rediseñar la estructura general del cliente.

\subsection{Arquitectura}

El frontend mantiene la organización por capas ya utilizada en el \acrshort{tfg}. El componente \textit{QueryApi} engloba la comunicación con el backend y con servicios externos, mientras que los componentes de interfaz se organizan por funcionalidad. Esta estructura no ha cambiado de forma significativa, pero ha permitido añadir nuevas vistas de forma incremental sin afectar al resto de la aplicación.

  \centering
  \includegraphics[width=0.95\textwidth]{imaxes/c4/c4_container_frontend.png}
  \caption{Diagrama C4 de contenedores del frontend}
  \label{fig:c4-container-frontend}
\end{figure}

En la el diagrama de arquitectura del frontend \ref{fig:c4-container-frontend} se representa como un contenedor React con su capa de estado, acceso a servicios, internacionalización y componentes de interfaz. También se observa su interacción con el backend y con los servicios externos utilizados para mapas.

En la parte inferior está el método de acceso a la \acrshort{api} del backend, implementado con \acrshort{rtkquery}. Encima se sitúan los componentes de interfaz, construidos con React y Material UI. El estado de sesión y el tema visual se guardan en Redux, mientras que la internacionalización se resuelve con \textit{react-i18next}. La configuración global de rutas se concentra en \textit{App.jsx}, donde se protegen pantallas según el estado de autenticación y el rol de usuario.

El login guarda el token en el estado global y en \textit{localStorage}, lo que permite restaurar la sesión al recargar la aplicación. Esa misma base de diseño facilita que las nuevas funcionalidades se integren como rutas o features adicionales sin modificar el esqueleto principal.

\subsection{Capa de acceso al servicio}

La capa de acceso a servicios está construida sobre una instancia base de \textit{createApi}. Esa base añade automáticamente el token JWT y el idioma activo en las cabeceras. A partir de ella se inyectan endpoints concretos para emergencias, asignaciones, organizaciones, usuarios, avisos, clima y reglas de recomendación.

Este enfoque reduce repetición y hace que cada feature declare solo la parte que necesita. Además, las mutaciones invalidan etiquetas para refrescar de forma automática las vistas dependientes. En el TFM esta capa se ha mantenido prácticamente igual en cuanto a estructura, pero se ha ampliado con nuevos endpoints asociados a asignaciones, reglas de recomendación y pantallas móviles, lo que ha sido posible gracias a la separación previa entre acceso a datos y componentes de interfaz.


\subsection{Capa de componentes de interfaz}

Como se puede ver en la imagen \ref{fig:c4-container-frontend}, la interfaz se divide por dominios funcionales. \textit{Dashboard} reúne el mapa, la información meteorológica y el panel de avisos, y se mantiene como una de las piezas centrales de la aplicación. \textit{LandingMap} dibuja cuadrantes y emergencias sobre MapLibre, con iconos distintos según el tipo de emergencia y recursos desplegados. \textit{Notice} y \textit{MyNoticesList} muestran avisos con sus imágenes y estados, y \textit{CoordinatorNoticesView} añade acciones de aceptación, rechazo y borrado.

La gestión de emergencias se divide en vistas como \textit{EmergencyManagementView}, \textit{EmergencyDetailScreen} y \textit{EmergencyHistoryView}. Las asignaciones disponen de vistas propias para creación, listado y detalle. La pantalla de creación de asignaciones permite seleccionar emergencia, cuadrante y recurso, y valida que la combinación sea coherente con la estructura de la emergencia. Del mismo modo, la vista de reglas de recomendación permite al coordinador seleccionar un tipo de emergencia, cargar las reglas asociadas, editar su prioridad y modificar el JSON que define el comportamiento del motor semiautomático.

En la práctica, el trabajo realizado en esta capa se ha centrado en ampliar componentes existentes y en añadir pantallas nuevas, no en cambiar la organización general. Por eso, el esfuerzo principal ha estado en refinar las vistas, integrar las nuevas funcionalidades y mantener el aspecto funcional con el frontend ya desarrollado en el \acrshort{tfg}.


\subsection{Internacionalización y librerías externas}

La internacionalización se basa en archivos de traducción para inglés, español y gallego. Los textos de la interfaz y los mensajes de error usan claves compartidas entre componentes, y el idioma activo se propaga en las peticiones al backend.

También se ha migrado a \textit{MapLibre}, una librería open source que permite prescindir de planes gratuitos o limitados. A mayores se usan librerías como \textit{Material UI} y \textit{dayjs}. Estas integraciones ya estaban presentes en el \acrshort{tfg} y se han mantenido porque encajan bien con la evolución del sistema y permiten incorporar nuevas funcionalidades sin rehacer la base del frontend.

\section{Subsistema Android}

\subsection{Objetivos}

La aplicación Android complementa al frontend web permitiendo consultar información operativa y ejecutar acciones frecuentes desde el móvil. Debe mostrar mapa, emergencias, organizaciones, perfil, avisos, equipo propio y asignaciones, además de soportar el envío de avisos con imagen y localización.

La interfaz móvil también debe adaptarse al idioma del usuario y mantener coherencia visual con la versión web, aunque con una navegación propia más adecuada al contexto táctil.

\subsection{Arquitectura}

La aplicación Android combina Compose con algunos componentes clásicos de Android cuando es necesario. \textit{MainActivity} actúa como contenedor principal y construye la navegación con \textit{NavHost}. El menú lateral cambia según el rol y muestra accesos a mapa, avisos, equipo, asignaciones, organizaciones y emergencias.

La configuración usa \textit{HttpClient} como capa de acceso a red. Esta clase centraliza las peticiones HTTP, lee el token desde preferencias compartidas y prueba varios hosts locales para facilitar el desarrollo. Además, la aplicación registra el token FCM del dispositivo para recibir notificaciones push.

La interfaz de Android mantiene componentes de Compose para pantallas nuevas, pero conserva fragments o activities en flujos donde la estructura existente ya los utiliza, como el envío de avisos o la visualización de detalles concretos.

\subsection{Capa de acceso al servicio}

Las peticiones se realizan mediante \textit{HttpClient}, que resuelve GET y POST contra varios hosts y añade la cabecera de autorización cuando existe sesión. Este diseño simplifica la reutilización y evita repetir código de conexión en cada pantalla.

Las pantallas consumen los mismos recursos del backend que el frontend web, por lo que la capa de acceso se limita a transformar el JSON recibido en estructuras manejables desde Kotlin.

\subsection{Capa de componentes de interfaz}

La pantalla principal se define en \textit{MainActivity} y su navegación agrupa cuatro bloques principales: mapa, avisos, equipo y gestión. \textit{MapScreen} usa MapLibre para mostrar cuadrantes activos y emergencias. \textit{EmergenciesScreen} lista emergencias filtrables y permite abrir su detalle. \textit{EmergencyDetailScreen} muestra la información de la emergencia, los cuadrantes vinculados y los recursos asociados.

La vista \textit{OrganizationsScreen} permite consultar organizaciones y aplicar filtros por nombre o tipo. \textit{ProfileScreen} muestra los datos del usuario y ofrece cambio de idioma y cierre de sesión. \textit{MyTeamScreen} presenta el equipo del usuario, sus miembros y el acceso a las asignaciones activas. \textit{MyAssignmentsScreen} lista las asignaciones pendientes y permite aceptarlas desde el propio dispositivo cuando el usuario dispone de permisos.

El envío de avisos se gestiona en \textit{SendNoticeFragment}, donde se combinan captura de imagen, obtención de localización y envío al backend. Esa pantalla refleja bien el objetivo de la app móvil: acercar al usuario acciones directas y relacionadas con su situación física.

\subsection{Internacionalización y librerías externas}

La internacionalización móvil se apoya en recursos de cadenas y en un helper propio para persistir el idioma activo. La aplicación también integra Firebase Messaging, MapLibre, Glide y Proj4J, además de utilidades para transformar coordenadas entre sistemas geográficos y proyectados.
